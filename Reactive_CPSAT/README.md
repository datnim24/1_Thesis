# Reactive CP-SAT Oracle

Offline oracle that answers: **"Given this disruption, what is the best the
factory could have achieved over the remaining shift?"**

## Concept

During a simulated shift, Unplanned Stops (UPS) randomly disable roasters. The
dispatching heuristic handles these events in real-time. The oracle, by contrast,
is an **offline benchmark**: after the simulation completes, it revisits each UPS
event and re-solves the scheduling problem from that disruption point using
CP-SAT (Google OR-Tools).

For each UPS event at time `t0`:

1. **Snapshot** — reads the simulation trace at slot `t0` to capture the exact
   factory state (RC/GC stock levels, roaster statuses, in-progress batches).
2. **Fixed intervals** — batches currently running on healthy roasters become
   immovable intervals (they will complete regardless).
3. **Downed roaster** — the disrupted roaster is blocked for `[t0, t0+duration)`.
4. **Re-solve** — builds a fresh CP-SAT model over the remaining horizon
   `[t0, 479]` and solves with a configurable time limit (default 2 min).
5. **Oracle\*(t0)** — reports the best achievable profit for this scenario.

This gives an upper bound on how well *any* reactive policy could perform at each
disruption point, enabling gap analysis against the heuristic.

## Key differences from CP_SAT_v2

| Feature | CP_SAT_v2 (deterministic) | Reactive Oracle |
|---------|--------------------------|-----------------|
| Horizon | Full shift [0, 480] | Remaining [t0, 479] |
| RC/GC init | From CSV | From simulation trace |
| Fixed intervals | None | In-progress batches |
| Downtime | Planned only | Planned + UPS block |
| RC stockout | Hard constraint (RC >= 0) | Soft penalty ($1500/event) |

## Usage

```bash
python -m Reactive_CPSAT.runner \
  --time 120 \
  --name "baseline_oracle_test" \
  --lambda-rate 2.0 \
  --mu-mean 20 \
  --seed 42 \
  --r3-flex 0
```

### CLI flags

| Flag | Default | Description |
|------|---------|-------------|
| `--time` | 120 | CP-SAT time limit per solve (seconds) |
| `--name` | "run" | Run name for output folder |
| `--lambda-rate` | 2.0 | UPS Poisson arrival rate |
| `--mu-mean` | 20.0 | Mean UPS duration (minutes) |
| `--seed` | 42 | Random seed |
| `--r3-flex` | 0 | R3 flexible routing (0 or 1) |
| `--input-dir` | "Input_data" | Input data directory |
| `--log-level` | INFO | Logging verbosity |

## Output

Results are saved to `Reactive_CPSAT/results/{YYYYMMDD_HHMMSS_{name}}/`:

- `reactive_result.json` — full result data (baseline KPIs, oracle results per
  UPS event, aggregate statistics)
- `report.html` — self-contained visual report with Gantt timelines
- `run.log` — full execution log

## Module structure

```
Reactive_CPSAT/
  __init__.py          Package init
  snapshot.py          Translate state.trace[t] + UPS -> d-dict for CP-SAT
  model.py             build_reactive(d) — adapted CP-SAT model
  oracle.py            ReactiveCPSATOracle class
  runner.py            CLI entry point (report via plot_result.py)
  results/             Output directory (git-ignored)
```

Reports are generated by the shared `plot_result.py` dashboard (Plotly-based),
which auto-detects `reactive_result.json` files and renders oracle-specific
panels (summary table, per-UPS Gantt charts, profit waterfall).
