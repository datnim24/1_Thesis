Nhà máy rang cà phê vận hành 1 ca = 480 phút. Có 5 máy rang (R1–R5) và 2 dây chuyền đóng gói (L1, L2). Mục tiêu là lên lịch rang để tối đa hóa Net Profit trong điều kiện có sự kiện mất điện ngẫu nhiên (UPS).

R1 L1 GC silo L1 PSC, NDG PSC=15, NDG=17 phút

UPS — Mất điện ngẫu nhiên

⚡ Unscheduled Power Stop: Tại bất kỳ slot nào, engine có thể fire UPS event cho bất kỳ roaster nào. Khi đó:
  • Nếu roaster đang RUNNING: batch bị hủy (GC đã trừ không hoàn lại, revenue=0), mto_remaining tăng lại nếu là MTO
  • Roaster chuyển sang DOWN với remaining = UPS_duration (ngẫu nhiên)
  • Pipeline của line đó bị reset về FREE
  • Sau khi DOWN hết timer → roaster về IDLE, needs_decision=True

2. Simulation Engine
Môi trường mô phỏng slot-by-slot (1 slot = 1 phút)

Engine chạy vòng lặp 480 slot. Mỗi slot thực hiện đúng 7 bước theo thứ tự cố định. Strategy (bất kỳ method nào) chỉ được gọi ở bước 6 và 7.

Vòng lặp 7 bước / slot

  1. UPS Events: Với mỗi UPS event có t == slot → hủy batch đang chạy, roaster → DOWN
  2. Roaster Timers: Mỗi roaster đang RUNNING/SETUP/DOWN: remaining -= 1. Nếu remaining == 0: RUNNING→IDLE + log batch + cộng revenue; SETUP→IDLE; DOWN→IDLE
  3. Pipeline + Restock Timers: pipeline_busy -= 1 cho mỗi line. Nếu mode==RESTOCK và pipeline_busy→0: gc_stock[pair] += restock_qty
  4. RC Consumption: Nếu slot này có consume event cho line L: rc_stock[L] -= 1. Nếu rc_stock[L] < 0: stockout_cost += $1,000
  5. Idle/Overflow Penalties: Mỗi roaster IDLE: nếu home_line RC < safety → idle_cost += $30; nếu out_line RC ≥ max_rc → over_cost += $50
  6. Restock Decision: Gọi strategy.decide_restock(state) → engine thực thi nếu hợp lệ
  7. Roaster Decisions: Với mỗi roaster có status==IDLE và needs_decision==True: gọi strategy.decide(state, roaster_id) → compute mask → apply action

SimulationState — toàn bộ trạng thái

status[rid] dict IDLE/RUNNING/SETUP/DOWN {R1:RUNNING, R2:IDLE, R3:DOWN, R4:RUNNING, R5:SETUP}

Action Masking — khi nào action được phép?

Trước khi gọi strategy.decide(), engine tính _compute_action_mask() trả về dict. Mọi action vi phạm bất kỳ điều kiện nào đều bị chặn cứng:

Điều kiện Action bị chặn Lý do kinh doanh
Roaster không IDLE Tất cả trừ WAIT Không thể bắt đầu khi đang bận
SKU không trong R_elig_skus[rid] Action đó R3 không làm NDG/BUSTA
gc_stock[pair] < 1 SKU cần GC đó Hết nguyên liệu
t + roast_time > 480 Action đó Batch không kịp xong trong ca
PSC: projected RC > max_rc PSC trên line đó Overflow nếu rang thêm
Batch overlap downtime slot Action đó UPS scheduled sẽ cancel batch
pipeline_busy > 0 (no setup) Immediate start Pipeline đang bận transport

📌 Ví dụ: Compute mask tại t=440, R1 IDLE

  • PSC L1: t+15=455≤480 ✅, gc(L1,PSC)=3≥1 ✅, projected RC L1=16≤40 ✅ → allowed
  • NDG: t+17=457≤480 ✅, gc(L1,NDG)=0 → blocked (hết GC)
  • BUSTA: R1 không eligible BUSTA → blocked
  • WAIT: luôn allowed
  • → mask = {("PSC","L1"):True, ("NDG",):False, ("WAIT",):True}

Projected RC — Tính toán overflow prevention

projected_RC(line, completion_t) = rc_stock[line] + running_PSC_completions(line, before completion_t) ← các batch đang chạy xong trước đó - future_consumption_events(line, now < t ≤ completion_t) ← consume events sẽ xảy ra + 1 ← batch mới sẽ thêm vào Nếu projected_RC> max_rc → block action (sẽ gây overflow)

3. Dispatch Heuristic
Rule-based baseline — không cần training, chạy mỗi decision point

Kiến trúc 2 layer

Layer 1: Restock (Global)

Mỗi phút kiểm tra xem có cần bổ sung GC không. Dùng Reorder Point (ROP):

ROP = ⌈(lead_time + 1) × eligible_roasters / roast_time⌉

Nếu gc_stock[pair] ≤ ROP → trigger restock

Layer 2: Roaster (Per-Roaster)

Mỗi lần roaster về IDLE:

  1. Check MTO urgency ratio ≥ 0.7?
  2. Nếu có → rang MTO (ưu tiên theo remaining)
  3. Không → rang PSC (R3 load-balance)
  4. Không khả thi → WAIT

📌 ROP Example: L1 PSC tại t=200

  • lead_time = 14 phút (restock_duration)
  • eligible roasters cho L1 PSC = R1, R2 → 2 roasters
  • roast_time PSC = 15 phút
  • ROP = ⌈(14+1)×2/15⌉ = ⌈2.0⌉ = 2
  • gc_stock(L1,PSC) = 1 ≤ 2 → kích hoạt restock ngay

📌 Urgency Example: NDG job tại t=380

  • NDG remaining = 2 batch, due = t=430
  • Time cần: 2×17 = 34 phút rang + 1 setup (nếu cần) = 39 phút
  • Time còn lại: 430-380 = 50 phút
  • urgency_ratio = 39/50 = 0.78 ≥ 0.7 → URGENT
  • → Dispatch ưu tiên R1 hoặc R2 vào NDG ngay

4. Q-Learning (Tabular)
Bảng Q-value — học từ kinh nghiệm, không dùng neural network

State Encoding — Tuple hashable

Q-Learning không thể xử lý state liên tục → phải rời rạc hóa (discretize) thành tuple có thể làm key cho dict: 

state_key = (
    "roaster",          # loại decision
    roaster_id,         # "R1".."R5"
    time_bin,           # 7–9 buckets anchored vào MTO due windows
    last_sku_code,      # 0=PSC, 1=NDG, 2=BUSTA
    pipe_busy,          # 0 hoặc 1
    rc_bin_home,        # 5 levels: ≤0 / ≤safety/2 / ≤safety / <max / ≥max
    rc_bin_other,       # như trên nhưng cho line kia (chỉ R3)
    urgency_bin,        # 5 levels: done/comfortable/watch/urgent/overdue
    gc_bin_eligible,    # 5 levels: 0 / 1 / <5 / <10 / ≥10
    setup_flag,         # 1 nếu vừa xong SETUP, 0 otherwise
)
5-level RC bins
Bin Điều kiện Ý nghĩa
0 rc ≤ 0 Stockout đang xảy ra
1 rc ≤ safety/2 Nguy hiểm
2 rc ≤ safety Thấp, cần rang thêm
3 rc < max_rc Bình thường
4 rc ≥ max_rc Đầy, không rang thêm
5-level Urgency bins
Bin Điều kiện Ý nghĩa
0 Không có MTO Done / no job
1 slack > 2×roast_time Thoải mái
2 slack > roast_time Cần theo dõi
3 slack ≤ roast_time Khẩn cấp
4 slack ≤ 0 Đã trễ deadline

Q-Update Formula

Q(s, a) ← Q(s, a) + α × [ r + γ × max_{a'} Q(s', a') − Q(s, a) ] α = 0.1–0.3 (learning rate — bước học) γ = 0.95–0.99 (discount factor — tầm nhìn tương lai) r = immediate reward (Δ net_profit) s' = state sau khi thực hiện action

📌 Ví dụ Q-update đầy đủ

  • State s = ("roaster","R2",5,0,0,2,0,3,2,0) → tức là: R2 IDLE, giữa ca, last=PSC, RC thấp, MTO urgent
  • Agent chọn action NDG (ID=2)
  • Reward r = +$7,000 (NDG hoàn thành) − $800 (setup) = +$6,200
  • Q(s, NDG) hiện tại = 150; max Q(s', a') = 800; α=0.1, γ=0.97
  • Q(s, NDG) ← 150 + 0.1 × [6200 + 0.97×800 − 150]
  • = 150 + 0.1 × [6200 + 776 − 150] = 150 + 0.1 × 6826 = 682.6 → 683
  • → Lần sau gặp state này, agent sẽ ưu tiên NDG vì Q cao
Hạn chế lớn nhất: Q-table chứa hàng triệu state nhưng chỉ 1 phần nhỏ được thăm. State chưa từng thấy → Q=0 → random decision. Không thể generalise: 2 state "giống nhau" nhưng encode khác → Q riêng biệt hoàn toàn. Đây là lý do cần Neural Network (DDQN).

5. RL-HH (Reinforcement Learning Hyper-Heuristic)
Dueling DDQN chọn 1 trong 5 heuristic tools mỗi decision point
Ý tưởng cốt lõi: Thay vì RL quyết định trực tiếp "rang PSC hay NDG hay WAIT?", RL-HH quyết định "dùng chiến lược nào?". Có 5 chiến lược (tools) được viết tay. Agent học khi nào nên dùng tool nào. Action space thu nhỏ từ 20+ xuống còn 5.

Quyết định xảy ra khi nào?

Mỗi tool chỉ cho ra 1 quyết định tại 1 thời điểm — không phải cả ca!
Cụ thể: mỗi khi một roaster về IDLE (sau khi batch xong, setup xong, hoặc UPS recovery), engine gọi strategy.decide(state, roaster_id). Đây là 1 "decision point". Tại decision point đó:
  1. Build observation vector (33 features)
  2. Chạy tất cả 5 tools để biết tool nào feasible và trả về action gì
  3. DDQN forward pass → Q[5] values
  4. Apply feasibility mask → argmax → Tool k
  5. Tool k trả về 1 action cụ thể (VD: ("PSC","L2"))
  6. Engine thực thi action đó cho roaster đó tại phút đó
Nếu có 5 roasters cùng về IDLE trong 1 phút → engine gọi decide() 5 lần → 5 tool selections độc lập.

Network groups — tại sao chia 3 branch?

Branch A — Roaster (15)

obs[1:16]: status + timer + last_sku của 5 roasters. Branch này học "roasters đang ở trạng thái gì?" để biết capacity còn lại.

→ Dense(15→64→32, ReLU)

Branch B — Inventory (11)

obs[16:27]: RC stocks + MTO remaining + pipeline modes + GC stocks + restock. Branch này học "inventory đang ở mức nào?"

→ Dense(11→64→32, ReLU)

Branch C — Context (7)

obs[0]+[27:33]: time + one-hot context. Branch này học "đây là thời điểm nào trong ca và quyết định cho ai?"

→ Dense(7→16, ReLU)

3 branch → concatenate (32+32+16=80) → merge(128, ReLU) → Value stream (→1) + Advantage stream (→5 tools)

Tại sao cần context one-hot?

Cùng state nhà máy nhưng: nếu đang hỏi R1 thì tool 0 trả về PSC→L1; nếu đang hỏi R3 thì tool 0 trả về PSC→L1 hoặc L2 (tùy balance). Cùng đó nếu là restock context thì tool 1 mới khả thi. Network cần biết đang ra quyết định cho ai để Q-values có ý nghĩa đúng.

7. 5 Tools — Chi tiết từng heuristic

Mỗi tool là 1 hàm Python nhận (state, roaster_id) và trả về action_id hoặc None (nếu không áp dụng được). Tool chỉ tính 1 action cho 1 decision point duy nhất — không ảnh hưởng tới các phút tiếp theo.

Tool 0 PSC_THROUGHPUT Roaster decisions only

Mục đích: Tối đa hóa thông lượng PSC. Luôn rang PSC ngay khi có thể. R3 được routing thông minh dựa trên trạng thái 2 lines.

Logic từng bước (code thực tế)

def _psc_throughput(state, roaster_id, feasible):
    if roaster_id is None:
        return None   # chỉ cho roaster context

    candidates = PSC_ACTIONS[roaster_id] ∩ feasible
    # PSC_ACTIONS: R1=[0], R2=[1], R3=[2,3], R4=[4], R5=[5]
    # action 2 = R3→PSC→L1, action 3 = R3→PSC→L2

    if len(candidates) == 0: return None
    if len(candidates) == 1: return candidates[0]   # R1/R2/R4/R5: chỉ 1 line

    # Chỉ R3 có 2 candidates: chọn line nào?
    # 1. UPS-aware routing (Cycle 14):
    l2_down = R4==DOWN OR R5==DOWN
    l1_down = R1==DOWN OR R2==DOWN
    if l2_down and not l1_down: return 3  # force L2 support
    if l1_down and not l2_down: return 2  # force L1 support

    # 2. Tightest-headroom routing (Cycle 5):
    score_L1 = min(max_rc - rc_stock[L1], gc_stock[(L1,PSC)])
    score_L2 = min(max_rc - rc_stock[L2], gc_stock[(L2,PSC)])
    return 2 if score_L1 > score_L2 else 3   # tie → L2

📌 R3 routing: t=200, state: R4=DOWN, RC(L1)=25, RC(L2)=10, GC_L1_PSC=15, GC_L2_PSC=8

  • l2_down = True (R4 DOWN), l1_down = False
  • → Force L2 support (R4 down nên L2 thiếu roaster) → return action 3 = R3→PSC→L2
  • Nếu không có UPS: score_L1 = min(40-25, 15) = min(15,15) = 15; score_L2 = min(40-10, 8) = min(30,8) = 8
  • → score_L1=15 > score_L2=8 → return action 2 = R3→PSC→L1

Feasible khi: Roaster eligible PSC + có PSC action trong feasible set (engine mask đã tính)

Return None khi: roaster_id=None (restock context) hoặc không có PSC action feasible

Tool 1 GC_RESTOCK Restock decisions only

Mục đích: Bổ sung GC silo khi sắp hết nguyên liệu. Không phải restock mọi lúc — chỉ khi thực sự cần thiết (tránh lãng phí pipeline time).

Logic từng bước

def _gc_restock(state, roaster_id, feasible):
    if roaster_id is not None:
        return None   # chỉ cho restock context!

    # Đếm roaster đang idle chờ từng line (Cycle 16: boost urgency)
    idle_L1 = count(r in {R1,R2} if status[r]==IDLE)
    idle_L2 = count(r in {R4,R5} if status[r]==IDLE)

    best_action = None
    best_ratio = +∞
    for each feasible restock action (line, sku):
        cap = gc_capacity[(line, sku)]
        stock = gc_stock[(line, sku)]
        ratio = stock / cap

        # Urgency boost: roaster đang đợi = thêm pressure
        ratio -= 0.05 × idle_per_line[line]

        # Threshold theo capacity:
        if cap >= 20 (PSC large silo):   threshold_ratio=0.35, threshold_abs=6
        else (NDG/BUSTA small silo):     threshold_ratio=0.40, threshold_abs=3

        if ratio >= threshold_ratio OR stock > threshold_abs:
            continue   # chưa cần restock
        if ratio < best_ratio:
            best_action = action; best_ratio = ratio

    return best_action  # None nếu không silo nào cần

Tool 2 MTO_DEADLINE Roaster decisions only

Mục đích: Đảm bảo MTO batch được làm kịp deadline. Ưu tiên job có nhiều batch nhất còn lại. Tie-break: BUSTA > NDG (BUSTA constrained hơn vì chỉ R2 eligible).

Logic từng bước

def _mto_deadline(state, roaster_id, feasible):
    if roaster_id is None: return None
    options = MTO_ACTIONS[roaster_id]
    # MTO_ACTIONS: R1=[(6,"NDG")], R2=[(7,"NDG"),(8,"BUSTA")]
    # R3/R4/R5: không có MTO actions → return None

    candidates = []
    for (aid, sku) in options:
        if aid not in feasible: continue
        remaining = sum(mto_remaining[jid] for jid where job_sku[jid]==sku)
        if remaining <= 0: continue
        priority = 1 if sku=="BUSTA" else 0   # BUSTA > NDG
        candidates.append((remaining, priority, aid))

    if not candidates: return None
    candidates.sort(reverse=True)   # most remaining first, then BUSTA
    return candidates[0][2]         # action_id của winner

📌 R2 tại t=300: NDG remaining=1, BUSTA remaining=2

  • NDG option: remaining=1, priority=0 → (1, 0, aid=7)
  • BUSTA option: remaining=2, priority=1 → (2, 1, aid=8)
  • Sort descending: [(2,1,8), (1,0,7)] → winner = BUSTA (aid=8)
  • → Tool 2 trả về action 8 = R2→BUSTA
  • Lý do: 2 batch BUSTA còn lại quan trọng hơn 1 batch NDG, phải làm ngay

Return None khi: roaster_id=None (restock), R3/R4/R5 (không eligible MTO), hoặc không có MTO pending

Tool 3 SETUP_AVOID Roaster decisions only

Mục đích: Tránh chi phí setup $800 và 5 phút bằng cách tiếp tục cùng SKU. Có exception thông minh: R1/R2 đang giữ PSC mà có MTO → làm MTO ngay (chỉ 1 lần PSC→MTO thay vì PSC→MTO→PSC→MTO).

Logic từng bước

def _setup_avoid(state, roaster_id, feasible):
    if roaster_id is None: return None
    last = last_sku[roaster_id]

    if last in ("PSC", None):
        # Exception (Cycle 4): R1/R2 với MTO pending → làm MTO trước
        # Logic: nếu làm PSC trước, sau đó phải PSC→MTO (1 setup)
        #        nếu làm MTO ngay, chỉ PSC→MTO (1 setup tổng, tốt hơn)
        if roaster_id in ("R1", "R2"):
            mto = _mto_deadline(state, roaster_id, feasible)
            if mto is not None: return mto   # ưu tiên MTO

        return _psc_throughput(state, roaster_id, feasible)   # fallback PSC

    # last = NDG hoặc BUSTA → tiếp tục cùng SKU (0 setup cost!)
    aid = MTO_ACTIONS[roaster_id][last]
    if aid not in feasible: return None
    remaining = sum(mto_remaining for this sku)
    return aid if remaining > 0 else None

  • Tool 3: last=NDG → tiếp tục NDG (action 6)
  • Tiết kiệm: 5 phút setup + $800 (so với chọn PSC rồi quay lại NDG)
  • Khi NDG remaining=0 → return None → tool 3 bị masked → agent chuyển sang tool khác

📌 Ví dụ Exception: R2 last=PSC, BUSTA pending=3

  • last_sku[R2] = PSC, bình thường sẽ rang PSC tiếp (0 setup)
  • Nhưng: Tool 3 exception → thử MTO_DEADLINE trước → BUSTA remaining=3 → return BUSTA
  • Lý do: Nếu rang PSC xong rồi mới rang BUSTA: 2 setups (PSC→BUSTA, BUSTA→PSC) = $1,600
  • Nếu rang BUSTA ngay: 1 setup (PSC→BUSTA) = $800 → tiết kiệm $800

CP-SAT biết trước MỌI sự kiện: UPS xảy ra lúc nào, consume events lúc nào, tất cả deadlines. Không phải online reactive mà offline optimization. Đây là trần lý thuyết — RL methods phải reactive nên luôn kém hơn.

Warm-start từ Dispatch

CP-SAT không bắt đầu từ trống. Engine chạy Dispatch heuristic trước để có 1 feasible schedule làm starting point. CP-SAT nhận hint variables và bắt đầu từ đó, cải thiện trong time budget cho phép (mặc định 60 giây).

Key takeaways

RL-HH vượt Q-Learning vì:
  • Không bị curse of dimensionality (5 tools, không phải 20+ actions)
  • Neural network generalise sang state chưa thấy
  • Tools encode domain knowledge → sample-efficient hơn
  • Tool mask đảm bảo feasibility tự động
Paeng v2 period-based trade-off:
  • ✅ Giảm số quyết định → ổn định hơn
  • ✅ Multi-roaster coordination trong 1 action
  • ❌ Kém responsive với UPS (period cũ có thể sai)
  • ❌ SSU fallback cho non-matching roasters = không optimal
